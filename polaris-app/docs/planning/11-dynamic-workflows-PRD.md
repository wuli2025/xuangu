# 11 · 动态编排（Dynamic Workflows · 多智能体）PRD

> 对话框新增「动态编排」模式：把一轮请求当成**多智能体编排**——编排器把目标拆成
> N 个相互独立的子任务，用 Claude Code 自带的 `Task` 子代理**并行扇出**，每条流水线
> 走 **实现 → 对抗式校验 → 修复**，最后由编排器汇总成最终交付。
>
> 设计依据：`PolarisKB/conversations/.../dynamic-workflows-分析与实现.html`（效果分析与落地）。
> 原则：**不另造编排框架**，直接借 Claude Code 现成的子代理机制（该架构本身就是这个形状）。

状态图例：⬜ 未开始 · 🔧 进行中 · ✅ 完成

---

## 0. 背景与触发

用户给出一张「Dynamic Workflows」分析图：一个 orchestrator 扇出 N 条任务，每条任务跑
implementer → verifier → fixer，全部完成后汇总返回。结论是：

- **对「能拆成多块 + 每块做完能被检查」的任务效果很好**（并行省墙钟时间、对抗式 verifier 拦住
  「看起来对、实际错」的输出、可扩展）。
- 对「一条思路走到底的对话 / 单点问答」**别上**——纯烧钱。

诉求：把这个能力做进**对话框下方的功能区**（与「目标模式」「请教毛主席」并列的模式开关），
并要求**比较好的分配效果**（活儿怎么拆、谁干哪块要让用户看得清）。

---

## 1. 效果判断（决定何时该上）

| | 该上 ✅ | 不该上 ❌ |
|---|---|---|
| 任务形态 | 可拆分 + 每块可独立验证 | 强顺序依赖 / 拆不开的整体 / 单点问答 |
| 例子 | 批量改写、多维代码/安全审查、调研 fan-out、逐条数据或文档处理、需多方独立判断的决策 | 普通对话、对成本/延迟极敏感的实时交互 |

代价（必须诚实）：N×（实现+校验+修复）多阶段，token 是单轮的几倍到几十倍 → 模式默认**关闭**，
由用户显式开启；后端指令里也写了「先判断该不该扇出，别为拆而拆」。

---

## 2. 三个不可省的核心原语（落到注入指令里）

1. **Orchestrator（编排器）** — 把目标拆成 N 个相互独立、边界不重叠的子任务。
2. **Fan-out + 限流** — 用 `Task` 并行扇出（同一回复里多个 `Task` 调用并发），但**每批 6~8 个**，跑完再下一批，别打爆速率/额度。
3. **每条流水线 实现 → 对抗式校验 → 修复** — 校验子代理 prompt 写死「**默认有问题、主动证伪**」；高风险 2~3 个校验投票。阶段间用**结构化结论**交接；**pipeline > barrier**（快的别等慢的）；改同批文件要**隔离**后合并。

收尾：编排器汇总成连贯交付，并在末尾交代**分配效果**（拆几块、各块谁干、校验拦下并修了什么）。

---

## 3. 实现（全部 additive，与既有 mode 同构）

复用「目标模式 / 请教毛主席」的**模式开关 → prompt 注入**机制，不引入新进程模型：

**前端**
```
src/components/ChatPanel.vue   # 工具栏新增「动态编排」按钮(Workflow 图标) + orchestrateMode 开关
src/stores/chat.ts             # send opts 增 dynamicWorkflow，透传后端
src/tauri.ts                   # ChatSendArgs 增 dynamicWorkflow?: boolean
```

**后端**
```
src-tauri/src/chat.rs
  ChatSendArgs.dynamic_workflow: bool          # 新字段(默认 false)
  dynamic_workflow_directive()                 # 注入编排指令(对齐 §2)
  allowed_tools_for(perm, with_task)           # 动态编排时额外放行 Task
  spawn_on_host(..., with_task)                # 把开关透传到 --allowedTools
```

**关键点**：headless（`--print`，stdin=null）下没人能逐个点「同意」，若不在 `--allowedTools`
里显式放行 `Task`，编排器扇出子代理会卡在权限确认上 → 多智能体根本跑不起来。因此**仅在
动态编排开启时**把 `Task` 加进 allowedTools（其余 mode 不变）。

## 5. 验收

- 对话框工具栏出现「动态编排」按钮，可点亮/熄灭；点亮后发送，后端注入编排指令并放行 `Task`。
- 对**可拆分 + 可验证**的请求（如「把这批文件逐个改写并各自校验」），模型以编排器身份扇出
  多个 `Task` 子代理并行干、每条 实现→校验→修复，末尾汇总并说明分配方案。
- 对**普通问答**，模型遵循指令**不扇出**，直接作答（避免烧钱）。
- 模式默认关闭；不影响「目标模式 / 请教毛主席 / 深度搜索」既有行为（其余 mode 的 allowedTools 不变）。
